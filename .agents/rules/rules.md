## プロジェクト概要

商品コードを軸とした機材レンタル管理デスクトップアプリ（ローカル実行）。
機材の貸出・返却・ステータス照会・履歴管理を行う。

### 処理フロー

1. 機材マスタをExcelから一括インポート
2. 商品コードで検索 → ステータス照会
3. レンタル登録（部署名・氏名・開始日・終了日）
4. 返却処理 → ステータスを「返却済」に更新
5. レンタル履歴を保持・閲覧

### 使用技術

- Python 3.x + Flask（バックエンド）
- SQLite（データベース）
- HTML / CSS / JavaScript（フロントエンド）
- pywebview（デスクトップ表示）
- openpyxl（Excelインポート用）
- PyInstaller（.exe 配布用）
- Azure AI Document Intelligence（画像解析）
- python-dotenv（環境変数管理）

### 制約事項

- 単一PC・単一ユーザー利用
- データエクスポート機能は不要
- 外部ネットワーク接続は画像解析時（Azure API）のみ


## ディレクトリ構成

```
機材レンタル管理ソフト/

├── build.spec               # PyInstaller ビルド設定
├── requirements.txt         # 依存パッケージ
├── .agents/rules/rules.md   # プロジェクトルール
├── docs/                    # ドキュメント
│   ├── requirements.md      # 要件定義
│   ├── task.md              # タスク計画
│   ├── handoff.md           # 引継ぎメモ
│   ├── test_plan.md         # テスト計画
│   └── README.md            # 利用者向けドキュメント
├── src/                     # 実装コード
│   ├── app.py               # Flask + PyWebView アプリケーション
│   ├── database.py          # DB初期化・操作
│   ├── azure_ocr.py         # Azure AI Document Intelligence クライアント
│   ├── templates/           # HTMLテンプレート
│   └── static/              # CSS / JS
├── data/                    # DB・インポート用ファイル（Git管理外）
│   └── rental.db            # SQLiteデータベース
└── dist/                    # ビルド生成物（Git管理外）
    └── 機材レンタル管理.exe   # 実行ファイル
```

### ディレクトリルール

- `src/` 配下にのみ実装コードを配置する。ルート直下へのスクリプト配置は禁止（`build.spec` は例外）
- `docs/` にはタスク管理ファイル・入力データのみ配置する
- `dist/` はビルド生成物の出力先とし、Git管理外とする
- 新規ディレクトリの追加時は、本ルールファイルの構成図も更新すること

## AI駆動開発フロー（エージェント行動指針）

### 1. プロジェクト初期構築
- `.agent/rules/rules.md`（コーディングルール）を作成する。
- `docs/requirements.md`（要件定義）を作成し、ゴールを明確化する。
- `docs/task.md`（タスク計画）を作成し、30分以内でテスト可能な最小単位に分割する。

### 2. 実装とサブエージェント活用
- `task.md`に沿って1タスクずつ実装と動作確認を行う。
- コンテキスト節約のため、特定機能の独立した実装は別チャット（サブエージェント）に委譲し、完成コードのみメインに合流させる。

### 3. テストとドキュメント同期
- 実データによるE2Eテストを実行し、実測値で検証する。
- リファクタリングを実施する。
- 実装過程で確定した詳細仕様を`requirements.md`に反映する。

### 4. 状態保存と引継ぎ
- 新規チャット移行前に、現在の進捗・未解決課題・次タスクを`docs/handoff.md`に要約する。
- 新規チャットの初回で`handoff.md`を読み込み、コンテキストを復元して開発を継続する。

## 5. 利用者向けドキュメント作成
- - `docs/README.md`を作成してセットアップ手順・使用方法・アーキテクチャ概要を記載する

## 6. 機能変更・拡張
- 仕様変更時は、まず`requirements.md`を修正する。
- 新要件に基づき`task.md`を再生成し、実装フェーズ（手順2）へ戻る。

## コーディング規約

- Python: PEP 8 準拠
- 関数・変数名は英語のスネークケース
- HTMLテンプレート: Jinja2構文
- JavaScript: ES6+
- コメントは意図を記述（日本語可）

## テスト規約

- 実データによる手動E2Eテストで検証する
- 各機能実装後にブラウザで動作確認を行う

## 依存パッケージ管理

- `requirements.txt` で管理する
- 最小限の外部ライブラリのみ導入する（Flask, openpyxl, pywebview）

## セキュリティ

- ローカル専用のためユーザー認証は不要
- SQLインジェクション対策: パラメータバインディング必須
- ファイルアップロード: 拡張子チェック（.xlsx のみ許可）

## ドキュメント規約

- `docs/` 内のドキュメントは Markdown で記述する
- 設計変更時は関連ドキュメントも同時に更新する
